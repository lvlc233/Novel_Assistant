## 1. 系统架构概览

采用 **前后端分离** + **微内核插件化 (Microkernel)** + **BFF 代理模式** 的架构。

- **前端**: Next.js 14 (App Router) + Zustand + TailwindCSS + Radix UI + Tiptap
- **后端**: FastAPI + SQLModel + PostgreSQL (pgvector)
- **Agent**: LangGraph + LangChain 1.1+（create_agent 与中间件）
- **插件系统**: 
    - **配置中心**: 管理插件元数据、开关状态、鉴权配置。
    - **BFF 数据管道**: 统一代理接口 `/api/v1/plugin/proxy/{plugin_id}/data`，实现数据源的重定向与标准化。
    - **UI 标准化**: 基于 `RenderType + Payload` 协议的通用渲染引擎。

### 架构原则
1. **配置与数据分离**: 插件表只存“配置/指针”，业务数据存储在官方表或第三方服务中。
2. **BFF 代理优先**: 前端不直接连接数据源，通过后端代理注入鉴权并清洗数据。
3. **协议标准化**: 所有插件通过标准接口（StandardDataResponse）与主系统交互。
4. **核心最小化**: 核心系统只负责路由分发和基础上下文维护。

### RenderType 与 Payload 协议
RenderType 只描述渲染策略，Payload 才是数据协议。不同 RenderType 对应不同的最小 Payload 结构。

| RenderType | Payload 最小结构 | 说明 |
| :--- | :--- | :--- |
| CONFIG | `fields: FieldNode[]` | 树形字段节点，支持嵌套与类型标注。 |
| AGENT_MESSAGES | `sessions: AgentSession[]` | 会话索引与摘要信息，详情通过会话ID拉取。 |
| CARD | `cards: CardItem[]` | 卡片列表或分组卡片，支持层级关系。 |
| LIST | `items: ListItem[]` | 标准列表项集合。 |
| DETAIL | `detail: DetailItem` | 单实体详情结构。 |
| DASHBOARD | `widgets: WidgetItem[]` | 指标卡片/图表单元集合。 |

## 插件概念
**核心理念：插件即数据管道 (Plugin as a Data Pipeline)**

我们重新定义了插件的角色：插件不再仅仅是“代码包”，而是一个**配置单元**。它负责描述“数据从哪里来 (Source)”以及“数据长什么样 (Schema)”。

### 1. 配置 vs 数据
- **配置 (Config)**: 存储在 `Plugin` 表中。包括：
  - `data_source_type`: 数据来源类型（如 `url` / `json` / `internal`）。
  - `data_source_config`: 与类型对应的配置（如 `url + headers` / `ns` / `data` / `service_id`）。
  - `auth_config`: 访问数据源所需的鉴权信息（如 API Key），由后端代理自动注入，前端不可见。
  - `render_type`: UI 渲染提示（如 `CONFIG`, `AGENT_MESSAGES`, `CARD`, `LIST`）。
- **数据 (Data)**: 存储在各自的业务表或外部服务中。
  - 插件本身不直接存储业务数据（如文章内容、对话记录），只通过 API 获取。

### 2. BFF 代理机制 (Backend for Frontend)
前端不直接请求 `data_source_url`，而是调用统一的代理接口：
`GET /plugin/proxy/{plugin_id}/data`

后端处理流程：
1. **鉴权**: 验证用户是否有权访问该插件。
2. **读取配置**: 获取插件的 `data_source_type`、`data_source_config` 和 `auth_config`。
3. **注入凭证**: 将 `auth_config` 中的敏感信息（如 Token）注入请求 Header。
4. **转发请求**: 按 `data_source_type` 选择读取器（URL/JSON/Internal）。
5. **数据清洗 (可选)**: 将上游返回的数据转换为标准 `StandardDataResponse` 格式（如果上游未遵循标准）。
6. **响应前端**: 返回标准化的数据包。

### 3. 作用域与优先级
（保留原有的作用域概念：Global / Work / Document）
- **结构定义**: 插件的基础元数据。
- **数值定义**: 运行时动态获取的数据内容。

---
通用前缀: `/api/v1`，以下路径皆省略前缀。

- 统一响应包装: `{code:int,data:T,message:str}`，SSE 流式除外
- 成功时: `code = 200`
- 非业务错误: `code = 50000`
- 业务错误码: `5 位数`
- HTTP status 按标准语义返回，`code` 作为业务分支
- 统一时间格式: ISO-8601
- 对外接口只出现 `*Request/*Response`，内部模型禁止直接作为 Response 字段

## 7. 统一数据库设计 (Database Schema)

> **说明**: 
> 1. 本项目使用 `SQLModel` (基于 SQLAlchemy + Pydantic) 定义数据库模型。
> 2. Agent 的运行状态（State/History）由 **LangGraph Checkpointer** 自动托管（通常涉及 `checkpoints`, `checkpoint_writes` 等表），**无需**在此处手动定义。
> 3. 这里的模型主要关注：业务实体、插件配置、资源关系以及“信箱”通知系统。

class PluginSQLEntity(SQLModel, table=True):
    """
    插件注册表: 定义系统中所有可用的插件（能力）。
    """
    __tablename__ = "plugin"

    id: UUID = Field(default_factory=create_uuid, primary_key=True, description="插件ID")

    # 全局开关
    enabled: bool = Field(default=True, description="全局启用状态")
    
    # 配置定义 (Schema) 与 默认配置
    # 存储 JSON 数据，例如 {"api_key": "string", "broadcast": "boolean"}
    config_schema: Dict = Field(default={}, sa_column=Column(JSON), description="配置Schema定义")
    default_config: Dict = Field(default={}, sa_column=Column(JSON), description="默认配置值")
    
    tags: List[str] = Field(default=[], sa_column=Column(JSON), description="标签列表")

    create_at: datetime = Field(default_factory=get_now_time, sa_type=TIMESTAMP(timezone=True))
    update_at: datetime = Field(default_factory=get_now_time, sa_type=TIMESTAMP(timezone=True))
    
    # Relationships
    work_mappings: List["WorkPluginMappingSQLEntity"] = Relationship(back_populates="plugin")

ps: 关于知识库和记忆插件之间的差距
1. 系统中的任何的Agent对知识库都只有读的操作,而没有主动写入知识库的权限。
2. 系统中的任何的Agent对记忆都只有被动加载而无主动读的功能,但是可以自行写入记忆。
3. 知识库面向的用户可以配置,记忆面向的是Agent可以配置,尽管对于用户都说都是可读写的。
4. 知识库是作品级别的,也就是不同作品之间有知识库隔离,而记忆是全局的,所有的Agent都可以访问到,在不同的作品中都生效。

## 插件类型
一共有3种,系统,官方,自定义级别
系统级的表示 例如,Agent管理,作品类型管理这种偏元管理的插件,其实更加准确的表达应该是系统级别的设置,包括语言,颜色快捷键...这些->不可以关闭
官方级的表示 例如,文档创作助手,项目助手这种官方提供的插件,->可开关或修改数值,但是不能修改结构
自定义级的表示 例如,用户自己定义的插件。->可以开关,修改数值,也可以修改结构。
一个有3个枚举
SYSTEM:系统
OFFICIAL:官方
CUSTOM:自定义

同时也有三个作用范围等级:(表示的其实是生命周期的概念)
分别是
全局: 针对整个项目都生效的:(也就是只要项目开着,那么就会生效)
作品: 针对该作品完整生效: (表示同一个作品的该插件的配置是相同的,但是不同的作品之间就不同的)
文档: 针对该文档完整生效: (表示同一个文档的该插件的配置是相同的,但是不同的文档之间就不同的)

当前的全局插件有

    "项目助手",
    "Agent管理插件",//SYSTEM
    "作品类型管理",//SYSTEM 

]
当前的作品插件有

当前的文档插件有

    "文档创作助手"

每个插件,除了SYSTEM级别的,都可以在作品中进行配置,其中,
当作为等级为: 全局,可以选择开启或关闭,但不能修改数值,表示该插件将会不会对该作品生效: 例如记忆,开启时,在该作品中进行交互时候,会进行自动的收集和总结生成,但是关闭的话,在该作品上的操作就不会收集了
当作用等级为: 作品, 可以选择开启或关闭,或调整数值,
当作用等级为: 文档, 可以选择开启或关闭,或调整数值,其效果相当于对该作品下的每个文档的配置都提供默认的配置,但是会被文档那级别的设置覆盖。
